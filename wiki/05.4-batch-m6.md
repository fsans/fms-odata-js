# Batch — $batch Multipart Requests (M6)

`fms-odata-js` supports the OData `$batch` protocol, allowing multiple reads and atomic write groups to be sent in a single HTTP round-trip. This reduces network latency and enables atomicity for write operations. **M6 is complete.**

---

## Quick Start

```ts
const batch = db.batch()

// Queue a read (GET)
batch.add({
  op: 'list',
  entitySet: 'contact',
  query: { $top: 10, $filter: "status eq 'active'" },
})

// Queue an atomic write group — all succeed or all fail together
batch.changeset(cs => {
  cs.create('contact', { first_name: 'Alice', last_name: 'Liddell' })
  cs.patch('contact', 42, { status: 'archived' })
  cs.delete('invoice', 99)
})

const result = await batch.send()

if (result.ok) {
  console.log('All operations succeeded')
}

// Per-operation results in request order
for (const r of result.responses) {
  console.log(r.status, r.ok, r.body)
}
```

## API Reference

### `FMSOData#batch()`

Returns a new `Batch` builder instance.

**Returns:** `Batch`

---

### `Batch#add(op)`

Queues a read operation (GET entity-set) to the batch.

**Parameters:**

| Param | Type | Description |
| :--- | :--- | :--- |
| `op.entitySet` | `string` | Entity set name (table / layout). |
| `op.op` | `'list'` | Operation type (currently only `'list'`). |
| `op.query` | `object?` | Optional query options: `$top`, `$skip`, `$filter`, `$select`. |

**Returns:** `BatchHandle<T>` — carries `_index` (position in response array) and `_promise` (resolves when batch settles).

---

### `Batch#changeset(build)`

Defines an atomic write group. All operations within succeed or fail together.

**Parameters:**

| Param | Type | Description |
| :--- | :--- | :--- |
| `build` | `(cs: Changeset) => void` | Callback that adds write operations to the changeset. |

---

### `Changeset#create(entitySet, body)`

Adds a POST (create) operation.

| Param | Type | Description |
| :--- | :--- | :--- |
| `entitySet` | `string` | Target entity set. |
| `body` | `Record<string, unknown>` | Field values for the new record. |

**Returns:** `BatchHandle<T>`

---

### `Changeset#patch(entitySet, key, body, opts?)`

Adds a PATCH (update) operation.

| Param | Type | Description |
| :--- | :--- | :--- |
| `entitySet` | `string` | Target entity set. |
| `key` | `string \| number \| boolean` | Primary key value. |
| `body` | `Record<string, unknown>` | Fields to update. |
| `opts.ifMatch` | `string?` | Optional `If-Match` header value for optimistic concurrency. |

**Returns:** `BatchHandle<T>`

---

### `Changeset#delete(entitySet, key, opts?)`

Adds a DELETE operation.

| Param | Type | Description |
| :--- | :--- | :--- |
| `entitySet` | `string` | Target entity set. |
| `key` | `string \| number \| boolean` | Primary key value. |
| `opts.ifMatch` | `string?` | Optional `If-Match` header for optimistic concurrency. |

**Returns:** `BatchHandle<void>`

---

### `Batch#send(opts?)`

Serialises the batch into a `multipart/mixed` body, POSTs to `/<db>/$batch`, and parses the multipart response.

**Options** (`RequestOptions`):

| Option | Type | Description |
| :--- | :--- | :--- |
| `signal` | `AbortSignal?` | Cancellation signal forwarded to `fetch`. |

**Returns:** `Promise<BatchResult>`

---

## Type Reference

### `BatchResult`

| Property | Type | Description |
| :--- | :--- | :--- |
| `responses` | `BatchOpResult[]` | All per-operation results in request order. |
| `ok` | `boolean` | `true` when every response has `status < 400`. |

---

### `BatchOpResult<T>`

| Property | Type | Description |
| :--- | :--- | :--- |
| `status` | `number` | HTTP status code of this individual operation. |
| `ok` | `boolean` | `true` when `status >= 200 && status < 300`. |
| `body` | `T \| undefined` | Parsed JSON body (or raw string for non-JSON responses). |
| `headers` | `Headers` | Response headers for this operation. |

---

### `BatchHandle<T>`

Returned by `Batch#add()` and `Changeset#create/patch/delete()`.

| Property | Type | Description |
| :--- | :--- | :--- |
| `_index` | `number` | Position of this operation in `BatchResult.responses`. |
| `_promise` | `Promise<T>` | Resolves with the operation body when the batch settles; rejects if the changeset fails. |

## Data Flow

```mermaid
sequenceDiagram
    participant App
    participant Batch as Batch (src/batch.ts)
    participant FMS as FileMaker Server

    App->>Batch: db.batch()
    App->>Batch: batch.add({ op: 'list', entitySet: 'contact' })
    App->>Batch: batch.changeset(cs => cs.create('contact', {...}))
    App->>Batch: batch.send()

    Batch->>Batch: _serialize() → multipart/mixed body
    Batch->>FMS: POST /<db>/$batch\nContent-Type: multipart/mixed; boundary=batch_<uuid>
    FMS-->>Batch: 200 OK (multipart/mixed response)

    Batch->>Batch: _parseResponse() → BatchOpResult[]
    Batch-->>App: BatchResult { responses, ok }
```

## Wire Format

The batch body is a standard `multipart/mixed` document. Read operations are top-level parts; changesets are nested `multipart/mixed` sub-documents.

```text
--batch_<uuid>
Content-Type: application/http
Content-Transfer-Encoding: binary

GET https://fms.example.com/fmi/odata/v4/DB/contact?$top=10 HTTP/1.1
Accept: application/json

--batch_<uuid>
Content-Type: multipart/mixed; boundary=changeset_<uuid>

--changeset_<uuid>
Content-Type: application/http
Content-Transfer-Encoding: binary

POST https://fms.example.com/fmi/odata/v4/DB/contact HTTP/1.1
Content-Type: application/json

{"first_name":"Alice","last_name":"Liddell"}
--changeset_<uuid>--

--batch_<uuid>--
```

## Implementation Notes

- **Boundary generation**: `crypto.randomUUID()` (Node 18+ / modern browsers), with a `Date.now()` + `Math.random()` fallback.
- **Query string encoding**: OData `$`-prefixed params (`$top`, `$filter`, `$select`, `$skip`) keep their literal `$` — `encodeURIComponent` is applied only to the values, not the names.
- **Response parsing**: Each MIME part's outer `Content-Type: application/http` / `Content-Transfer-Encoding: binary` headers are stripped before parsing the inner HTTP status line (e.g. `HTTP/1.1 201 Created`) and body.
- **Atomicity**: If any operation inside a `Changeset` has `status >= 400`, all `BatchHandle` promises for that changeset reject with `Changeset failed: <status>`. Individual reads are resolved independently.
- **String key escaping**: Single quotes in string keys are doubled (`O'Brien` → `contact('O''Brien')`), consistent with the rest of the library.

## Tests

14 unit tests in `tests/unit/batch.test.ts` covering:

- Serialisation: single read, multiple reads, changeset with create, changeset with PATCH + DELETE, string key quote escaping
- Response parsing: single response, multiple responses in order, error status (404), mixed read + changeset
- Request plumbing: `Content-Type: multipart/mixed; boundary=batch_…` header, `POST /<db>/$batch` URL
- Cancellation: `AbortSignal` forwarded to `fetch`
- Changeset options: `If-Match` header on PATCH

Live integration test in `tests/integration/live.test.ts`: sends a batch containing a top-1 read and a single-create changeset, asserts both responses are successful (`status >= 200 && < 300`), and registers any created rows in `createdContactKeys` for `afterAll` cleanup.
